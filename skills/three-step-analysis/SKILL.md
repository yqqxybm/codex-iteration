# 三步认真分析

## Codex 执行姿态

在三步分析中，Codex 的角色首先是深度分析者，而不是执行器；用户要求"分析 / 三步分析 / 思考 / 评估"时，不自动修改文件或执行命令。

这是深度思考的唯一认知内核。软件项目、代码、架构、调试、发布风险等项目语境，使用
`project-analysis` 作为适配层；适配层只能映射项目约束、验证和调用链，不另造一套三步哲学。只有用户明确要求"通用三步分析"或问题本身不是软件项目时，才直接按本 skill 执行。

只有用户明确要求"改 / 实现 / 修复 / 执行 / 部署 / 同步"，或请求本身是修复、实现、构建、部署、同步等执行类任务，才进入阶段 4。

当用户要求认真分析或解决某个问题时，按以下框架执行。这个框架融合了多种经过验证的推理方法（链式思维、树式探索、辩证反思、抽象提炼），目的是避免浅表分析，真正深入问题本质。

## 本体解释

三步分析不是三段回答，而是在不确定世界中建立可校准的判断模型：

- 阶段 1 建立问题世界模型：识别角色、目标、约束、反馈、风险、时间尺度和主导矛盾。
- 阶段 2 校准最可能导致误判的变量：只问会改变判断、边界、优先级或执行路径的问题。
- 阶段 3 在明确边界下作出可验证承诺：给出能被证据反驳或确认的判断、方案或计划。

它控制的是认知和承诺，不自动控制现实行动。只有进入阶段 4 时，才把判断转成执行。

## 前置：判断任务类型与复杂度

在开始之前，先做两个判断：

**任务类型：**
- **分析/方案类**（分析、评估、比较、诊断、给方案）：执行阶段 1 → 2 → 3
- **执行类**（明确要求执行、修复、实现、构建、部署、同步）：执行阶段 1 → 2 → 3 → 4

**复杂度自适应：**
根据问题的复杂度动态调整每个阶段的深度。判断依据包括：涉及的变量数量、不确定性程度、影响范围、是否存在明显的利益冲突或技术权衡。

- **高复杂度**（多变量、高不确定性、重大影响）：每个阶段都深入展开，阶段 1 探索不少于 3 个方向，阶段 2 先识别 4-6 个关键信息缺口，再压缩成 1-3 个真正阻塞的问题
- **中等复杂度**：标准深度，阶段 1 探索 2-3 个方向，阶段 2 识别 2-4 个信息缺口，并只提出会改变结论的 1-3 个阻塞问题
- **低复杂度**（变量少、约束清晰）：精简执行，每个阶段直奔要点，避免过度分析

这样做的原因是：对简单问题做冗长分析既浪费时间又模糊焦点；对复杂问题做浅层分析则遗漏关键因素。让分析深度匹配问题难度。

## 最优判断标准

阶段 3 的"最优"不是最新、最复杂或覆盖最多的方案，而是在当前证据和约束下达成反思性均衡：忠实服务用户意图，判断规则能稳定导出正确行为，复杂度由实际价值证明。只有当新方案暴露未解决的核心张力，或用更少结构保持同等/更好行为时，才替代当前判断。

## 阶段 1：专家头脑风暴

目标：全面理解问题，穷举相关因素，**并提炼问题的本质**。

这个阶段的关键在于：不要急于跳进细节。先后退一步（Step-back），从问题的底层原则和核心矛盾入手，再展开具体维度分析。这样做能避免"见树不见林"的陷阱。

### 执行步骤

**第一步：抽象提炼——这个问题的本质是什么？**

在列举任何细节之前，先回答：
- 这个问题的**底层原则**或**核心矛盾**是什么？
- 它属于哪一类问题？（资源分配、技术选型、风险管理、流程优化……）
- 什么因素最终决定了方案的好坏？（换句话说：评判标准是什么？）

这一步的价值在于：很多看似复杂的问题，其本质往往可以归结为几个核心权衡。识别这些权衡，后续分析才有方向。

**第二步：多维度展开**

围绕核心矛盾，从多个维度展开分析：
1. 列出问题涉及的**所有相关维度**（技术、业务、性能、安全、兼容性、成本、用户体验、可维护性等——根据问题性质选择相关维度，不需要机械列举所有维度）
2. 分析当前已知的上下文信息（现状、背景、依赖关系等）
3. 识别**潜在风险和约束条件**

**第三步：多路径探索**

列出不少于 2 个可能的解决方向。对每个方向，用一两句话说明其核心思路和适用条件。

这里借鉴树式思维（Tree-of-Thought）的理念：不要过早收敛到单一方案。即使某个方向看起来明显更优，也值得花几句话说明为什么其他方向不适合——这本身就是对推荐方案的一种验证。

**第四步：辩证检验——挑战自己的分析**

这一步很重要，也是多数分析框架容易遗漏的。主动扮演"反方"，检验自己的思路：
- 我的分析是否有**隐含假设**？这些假设成立的条件是什么？
- 如果一个持不同观点的专家看到我的分析，他最可能质疑哪一点？
- 是否存在我尚未考虑到的**反例**或**边界情况**？

这不是为了推翻自己的分析，而是为了发现盲区。如果辩证检验后分析仍然成立，说明它更可靠；如果发现了问题，及时修正比在后面返工好得多。

### 输出结构

```
### 阶段 1：专家头脑风暴

**问题本质：** [一两句话概括核心矛盾/底层原则]
**评判标准：** [什么决定了方案的好坏]

**多维度分析：**
- [维度1]: [分析]
- [维度2]: [分析]
...

**已知信息与约束：**
- [关键信息/约束]
...

**可能方向：**
1. [方向1]: [核心思路和适用条件]
2. [方向2]: [核心思路和适用条件]
...

**辩证检验：**
- 隐含假设: [识别的假设及其成立条件]
- 可能的质疑: [反方可能提出的观点]
- 修正/补充: [如有需要]
```

## 阶段 2：反向询问

目标：通过**引导式提问**补全信息盲区，确认用户意图和现实约束。

## Codex structured question rule

During Stage 2, use a **blocking calibration gate** when the user explicitly
asks for `三步分析` / `three-step-analysis`, names this skill/framework, or a
missing calibration would materially change the answer. For explicit framework
calls, at least one user-owned calibration question is presumed necessary before
Stage 3. The question should verify the goal, priority, constraint, judgment
standard, action boundary, or recommended path that would most change the answer
quality.

When a blocking gate is required, call `request_user_input` for 1-3 structured
questions if it is available. If `request_user_input` is unavailable, stop at
Stage 2, state the observed tool/mode evidence, and ask concise text questions
instead. Do not continue to Stage 3 until the user answers.

For explicit framework calls, only two cases may bypass the blocking
calibration gate:

1. **Skill-internal invocation**: the user is explicitly asking Codex to use a
   skill or thinking framework as part of another skill workflow, such as
   review, optimize, self-refine, or framework-pass validation. In that case,
   follow the owner skill's convergence and logging rules instead of forcing an
   extra user stop.
2. **Fact verification**: the task is primarily factual lookup, source
   verification, or current-state checking where the answer can be determined
   from evidence rather than user preference.

This rule applies only to three-step-analysis. For ordinary tasks, ask when
ambiguity or quality risk justifies it.

### 为什么这个阶段很关键

阶段 1 的分析再全面，也是基于已知信息的推理。而真实决策中，"不知道自己不知道什么"往往是最大的风险。这个阶段的目的是把那些隐藏的决定性因素挖出来。

### 执行要求

1. 基于阶段 1 识别的信息缺口和辩证检验中发现的不确定点，提出**针对性问题**；显式三步分析调用默认至少提出 1 个阻塞式校准问题，非显式触发只在缺口会改变结论时阻塞
2. 问题应覆盖：用户偏好/优先级、现实约束、预期目标、不确定的技术细节
3. 只问会改变判断、边界、优先级或执行路径的问题。不要为了使用提问工具而问无关偏好。
4. **引导式提问**：不要只列出开放问题让用户从零思考。对每个问题，给出你的**初步判断或选项**，让用户确认、修正或补充。这样做的好处是降低用户的认知负担，同时让用户知道你已经在思考，不是机械地列清单

   示例对比：
   - 差：「你的预算是多少？」
   - 好：「根据类似项目的经验，这类方案的成本通常在 X 到 Y 范围。你的预算大致在哪个区间？或者有其他约束？」

5. **阻塞式校准必须使用 Codex 的结构化提问工具**：如果工具可用，使用 `request_user_input`。如果工具不可用，停止在阶段 2，说明观察到的工具/模式证据，并用简短文字提问；不得继续阶段 3。
6. 为了尽可能高质量使用 `request_user_input`，先把信息缺口压缩成 **1-3 个阻塞问题**；每个问题给出 2-3 个互斥选项，并把推荐选项放第一位。每个问题必须包含稳定的 `id`、12 字以内的 `header`、以及带 `(Recommended)` 的推荐选项标签。
7. 显式三步分析调用中，只有 skill 内部调用或事实核查类任务可以豁免阻塞式校准；豁免时必须明确说明属于哪一类，并给出继续进入阶段 3 的证据边界。
8. **一旦提出阻塞问题，必须等待用户回答后才能进入阶段 3**——这是阻塞性的。
9. 对于高复杂度问题，可以分优先级提问：先问决定性因素，再问次要细节；不要为了使用工具而问无关偏好。

### 输出结构

```
### 阶段 2：反向询问

阻塞情况（显式三步分析调用，或关键缺口会改变结论）：

基于前面的分析，我需要先做一次阻塞式校准，避免把关键假设带入结论：

1. [问题1 - 关于核心判断/边界/优先级]
   → 我的初步判断是 [推荐选项]，因为 [理由]。请确认或修正。

豁免情况：

**阻塞校准豁免：** 当前属于 [skill 内部调用 / 事实核查]。证据边界是 [X]，下面直接进入阶段 3。

非阻塞情况：

**非阻塞校准：** 当前不是显式三步分析调用，且缺口不会改变结论；采用 [明确假设]，下面直接进入阶段 3。

多问题时：

1. [问题1 - 关于核心决策因素]
   → 我的初步判断是 [X]，因为 [理由]。这与你的情况一致吗？
2. [问题2 - 关于约束条件]
   → 常见的选择是 [A] 或 [B]，各有 [特点]。你倾向哪个？
3. [问题3 - 关于不确定的细节]
   → 这个会影响 [具体影响]。你目前的情况是？
...
```

## 阶段 3：计划制定

目标：综合所有信息，给出经过验证的最优方案。

### 执行步骤

**第一步：方案综合**

结合阶段 1 的多路径探索，以及阶段 2 用户确认的信息或明确假设：
- 如果存在一个明显最优方案 → 直接给出该方案及理由
- 如果存在多个各有优劣的方案 → 同时列出，并给出各自特点、适用场景和取舍建议
- 如果用户的回答改变了之前的假设 → 明确说明调整了什么以及为什么

**第二步：方案验证（Reflexion）**

在定稿之前，对方案做一次自检：
- **逻辑一致性**：方案的各个步骤之间是否有矛盾？
- **完备性**：是否覆盖了阶段 1 识别的所有关键维度？
- **可行性**：在用户确认的约束条件下，方案是否真的可执行？
- **风险覆盖**：阶段 1 中识别的风险，方案中是否有对应的应对措施？

如果自检发现问题，在输出中说明并调整，而不是假装没看到。

**第三步：输出方案**

方案必须包含：具体步骤、预期效果、风险及应对。对于执行类任务，步骤应该具体到可以直接执行。

### 输出结构

单一方案时：
```
### 阶段 3：计划

**推荐方案：** [方案名称]
**核心理由：** [为什么是最优——关联到阶段 1 的评判标准]

**具体步骤：**
1. [步骤1] — [预期结果]
2. [步骤2] — [预期结果]
...

**预期效果：** [描述]
**风险及应对：**
- [风险1] → [应对措施]
- [风险2] → [应对措施]

**方案自检：** [一两句话说明自检结果，或说明做了哪些调整]
```

多方案时：
```
### 阶段 3：计划

#### 方案 A：[名称]
- **核心优势：** [描述]
- **步骤概述：** [概述]
- **最适合的场景：** [描述]
- **主要代价/风险：** [描述]

#### 方案 B：[名称]
- **核心优势：** [描述]
- **步骤概述：** [概述]
- **最适合的场景：** [描述]
- **主要代价/风险：** [描述]

**对比与建议：** [基于用户确认的优先级，给出倾向性建议及理由]
```

## 阶段 4：实施（仅执行类任务）

目标：按最优方案执行，**并在过程中保持反馈循环**。

### 执行要求

1. 选择阶段 3 中的最优方案开始实施。如果阶段 3 给出了多方案且用户未明确选择，选择综合最优的方案并说明原因
2. **分步交付，边做边报告**：不要等全部做完再一次性展示。每完成一个关键步骤，简要说明进展和发现。这样做有两个好处：用户可以尽早发现偏差，避免大量返工；遇到意外情况时可以及时调整方向
3. 实施过程中遇到预期外问题时，及时说明并调整方案——不要硬着头皮按原计划走
4. 实施完成后进行**结果验证**：
   - 对照阶段 1 的评判标准，检查结果是否达标
   - 对照阶段 3 的预期效果，检查实际效果是否匹配
   - 如有偏差，说明原因和后续建议

## 原则与注意事项

- 每个阶段的输出清晰标注阶段编号和名称
- 显式三步分析调用的阶段 2 默认必须提出阻塞式校准问题，并等待用户回复后再继续；非显式触发只在关键校准会改变结论时阻塞；显式调用中只有 skill 内部调用或事实核查类任务可以豁免，豁免时必须明示原因
- **不要跳过辩证检验**——它是区分浅层分析和深度分析的关键
- 分析要有深度，但深度不等于冗长。说到点子上比面面俱到更重要
- 如果在后续阶段发现前面阶段的分析有误或不完整，**允许回溯修正**，并明确说明修正了什么。线性推进不是目的，高质量结论才是
- 输出格式是指引而非枷锁。如果某个问题的特性不适合上面的模板，可以灵活调整结构，但核心步骤（抽象提炼→多维分析→辩证检验→引导式提问→方案验证）不能省略
