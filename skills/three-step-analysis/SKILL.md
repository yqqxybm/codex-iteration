# 三步认真分析

## Codex 执行姿态

在三步分析中，Codex 的角色首先是深度分析者，而不是执行器；用户要求"分析 / 三步分析 / 思考 / 评估"时，不自动修改文件或执行命令。

这是深度思考的唯一认知内核。软件项目、代码、架构、调试、发布风险等项目语境，使用
`project-analysis` 作为适配层；适配层只能映射项目约束、验证和调用链，不另造一套三步哲学。只有用户明确要求"通用三步分析"或问题本身不是软件项目时，才直接按本 skill 执行。

只有用户明确要求"改 / 实现 / 修复 / 执行 / 部署 / 同步"，或请求本身是修复、实现、构建、部署、同步等执行类任务，才进入阶段 4。

当用户要求认真分析或解决某个问题时，按以下框架执行。这个框架融合了多种经过验证的推理方法（链式思维、树式探索、辩证反思、抽象提炼），目的是避免浅表分析，真正深入问题本质。

## 本体解释

三步分析不是三段回答，而是在不确定现实中形成可校准判断的认识方法。

它承认：现实不是直接透明的，判断会被局部材料、隐含前提和行动冲动扭曲，
答案也不能脱离实践成立。因此它先充分展开具体材料与可能思路，再抽象出
问题世界模型；再校准最可能导致误判的变量；最后在明确边界下给出可验证、
可执行、可反驳的承诺性判断。

它的主干是原哲学：在不确定世界中建立可校准的判断模型。现有增强只是
实现机制：默认判断、方案排序、无知地图、问题账本、结构化提问和实践验证
都必须服务于这个主干，不能反过来压过主干。

- 阶段 1 建立问题世界模型：先铺开相关信息、背景、约束、风险、参与角色、反馈回路、时间尺度和可能思考方向，再从材料中识别底层原则、核心矛盾、主导变量和评判标准。
- 阶段 1 必须产出可校准模型：默认判断、方案排序、关键假设、反转条件、可自行判断的信息、必须由用户确认的变量。
- 阶段 1 必须产出无知地图：已知、推断、未知、反转变量、可验证变量、用户私有变量和暂时不可知变量。
- 阶段 2 校准误判变量：审查当前判断在哪些条件下会错，所有关键问题都必须有去向，只有用户私有且会改变判断、边界、优先级或执行路径的问题才问用户。
- 阶段 3 作出可验证承诺：给出能被证据反驳或确认的判断、方案或计划，并说明如何在实践反馈中验证、暴露失败和进入修正。

它控制的是认知和承诺，不自动控制现实行动。只有进入阶段 4 时，才把判断转成执行。

## 前置：判断任务类型与复杂度

在开始之前，先做两个判断：

**任务类型：**
- **分析/方案类**（分析、评估、比较、诊断、给方案）：执行阶段 1 → 2 → 3
- **执行类**（明确要求执行、修复、实现、构建、部署、同步）：执行阶段 1 → 2 → 3 → 4

**复杂度自适应：**
根据问题的复杂度动态调整每个阶段的深度。判断依据包括：涉及的变量数量、不确定性程度、影响范围、是否存在明显的利益冲突或技术权衡。

- **高复杂度**（多变量、高不确定性、重大影响）：每个阶段都深入展开，阶段 1 探索不少于 3 个方向，并明确方案排序、关键假设、无知地图、反转条件和用户私有变量；阶段 2 生成问题账本，并只把其中真正阻塞的问题交给用户
- **中等复杂度**：标准深度，阶段 1 探索 2-3 个方向，并产出简短可校准模型和问题账本种子；阶段 2 治理关键问题，只提出会改变结论的用户私有反转变量
- **低复杂度**（变量少、约束清晰）：精简执行，每个阶段直奔要点，避免过度分析

这样做的原因是：对简单问题做冗长分析既浪费时间又模糊焦点；对复杂问题做浅层分析则遗漏关键因素。让分析深度匹配问题难度。

## 最优判断标准

阶段 3 的"最优"不是最新、最复杂或覆盖最多的方案，而是在当前证据和约束下达成反思性均衡：忠实服务用户意图，抓住并处理核心矛盾，判断规则能稳定导出正确行为，复杂度由实际价值和实践验证证明。只有当新方案暴露未解决的核心张力，或用更少结构保持同等/更好行为时，才替代当前判断。

一句话标准：三步分析的本质，是在不确定现实中，通过材料展开、结构抽象、
误判校准和实践承诺，把一个表层问题转化为当前条件下最可靠、最可执行、
最可验证的判断。

## 阶段 1：专家头脑风暴

目标：全面理解问题，穷举相关因素，**并提炼问题的本质**。

这个阶段的关键在于：不要急于给答案，也不要空转哲学。先把相关材料和
可能思考方向铺开，再后退一步（Step-back），从问题的底层原则和核心矛盾
入手，回到具体维度分析。这样做能避免"只见树不见林"，也避免没有材料支撑
的抽象。

### 执行步骤

**第一步：材料铺开——这个问题所在的现实是什么？**

在抽象之前，先列出足以支撑判断的材料：
- 用户的直接目标、隐含目标和当前语境
- 相关背景、历史决策、已有约束和现实边界
- 参与角色、利益关系、反馈回路和时间尺度
- 可能思考方向、可选路径、已暴露风险和潜在风险
- 哪些信息来自事实，哪些只是推断

这一步不是罗列凑数，而是防止过早抽象。没有材料铺开，后面的"本质"很容易
只是第一印象。

**第二步：抽象提炼——这个问题的本质是什么？**

在材料铺开的基础上回答：
- 这个问题的**底层原则**或**核心矛盾**是什么？
- 它属于哪一类问题？（资源分配、技术选型、风险管理、流程优化……）
- 什么因素最终决定了方案的好坏？（换句话说：评判标准是什么？）

这一步的价值在于：很多看似复杂的问题，其本质往往可以归结为几个核心权衡。识别这些权衡，后续分析才有方向。

**第三步：多维度展开**

围绕核心矛盾，从多个维度展开分析：
1. 列出问题涉及的**所有相关维度**（技术、业务、性能、安全、兼容性、成本、用户体验、可维护性等——根据问题性质选择相关维度，不需要机械列举所有维度）
2. 分析当前已知的上下文信息（现状、背景、依赖关系等）
3. 识别**潜在风险和约束条件**

多维度展开不能停留在罗列。展开后的结果必须回到第一步：判断哪个矛盾或
变量最支配答案好坏，哪些维度只是背景、约束或次要影响。

**第四步：多路径探索**

列出不少于 2 个可能的解决方向。对每个方向，用一两句话说明其核心思路和适用条件。

这里借鉴树式思维（Tree-of-Thought）的理念：不要过早收敛到单一方案。即使某个方向看起来明显更优，也值得花几句话说明为什么其他方向不适合——这本身就是对推荐方案的一种验证。

**第五步：辩证检验——挑战自己的分析**

这一步很重要，也是多数分析框架容易遗漏的。主动扮演"反方"，检验自己的思路：
- 我的分析是否有**隐含假设**？这些假设成立的条件是什么？
- 如果一个持不同观点的专家看到我的分析，他最可能质疑哪一点？
- 是否存在我尚未考虑到的**反例**或**边界情况**？

这不是为了推翻自己的分析，而是为了发现盲区。如果辩证检验后分析仍然成立，说明它更可靠；如果发现了问题，及时修正比在后面返工好得多。

**第六步：默认判断、无知地图与可校准模型——让好问题从模型里长出来**

阶段 1 不能只停留在"我分析了几个维度"。在进入阶段 2 前，必须形成一个可校准模型：

- **默认判断**：如果现在不问用户，基于已有证据我会给出什么结论。
- **方案排序**：当前为什么 A 优于 B，B 又优于 C；不能只并列罗列方案。
- **关键假设**：默认判断依赖哪些事实、价值排序、边界或授权。
- **反转条件**：哪个假设一旦错了，结论、方案排序或执行边界会改变。
- **可自行判断的信息**：哪些缺口可以通过上下文、文件、命令、网络事实或常识推断解决。
- **用户私有变量**：哪些反转条件只能由用户提供，例如真实目标、价值排序、现实约束、授权边界。

同时必须形成一张无知地图。好分析不只是知道什么，还要知道自己不知道什么：

- **已知事实**：来自用户、上下文、文件、命令或可靠来源。
- **推断事实**：Codex 基于经验或局部证据推断出来，但需要标明依据。
- **未知变量**：当前缺失的信息。
- **反转变量**：一旦不同会改变默认判断、方案排序或执行边界的未知项。
- **可验证变量**：可以通过查文件、跑命令、联网、实验或审查获得答案的问题。
- **用户私有变量**：只能由用户确认的目标、价值排序、现实约束或授权边界。
- **暂时不可知变量**：本轮无法可靠获得，只能作为风险、假设或后续验证项处理。

没有默认判断、无知地图、反转条件和用户私有变量审计，阶段 1 不算完成。阶段 2 的问题账本只能从无知地图和反转条件中产生，不能临时编问题。

## 阶段 2：反向询问

目标：校准最可能导致误判的变量，必要时通过**引导式提问**确认用户意图和
现实约束。本阶段的本质是审查当前判断的可错条件：问题必须服务于修正判断，
而不是补一个信息空格或把判断责任转移给用户。

## Codex structured question rule

Stage 2 is **question governance**, not an automatic question step. Explicit
`三步分析` / `three-step-analysis` calls must include Stage 2, and Stage 2 must
produce a compact `question_ledger` for every material unknown from Stage 1's
无知地图. For low-complexity tasks this ledger can be brief, but it cannot be
omitted. No key question may disappear without an action.

Each ledger item must be classified:

- `self-answerable`: Codex should answer by reasoning from context or existing
  evidence.
- `evidence-needed`: Codex should inspect files, run commands, search sources,
  test, or otherwise verify.
- `assumption-continue`: the uncertainty matters, but a reasonable assumption
  preserves progress; state the assumption and its risk.
- `user-private-blocking`: the answer is user-private and would change the
  conclusion, ranking, boundary, or execution path.
- `follow-up-risk`: the issue is real but not blocking this answer; carry it as
  a risk, validation item, or later question.

Use `request_user_input` only for `user-private-blocking` questions that meet
all conditions:

1. it is tied to a named **反转条件** from Stage 1,
2. the answer would change the conclusion, plan ranking, boundary, or execution
   path,
3. the variable is user-private and cannot be determined from context, files,
   commands, sources, or Codex's own judgment,
4. the question asks the user to confirm a real goal, value ordering, constraint,
   authorization boundary, or external state,
5. the question does not ask the user to do Codex's reasoning, choose between
   under-argued options, or decide whether Codex should proceed when the user
   already asked for analysis.

If a blocking question is justified and `request_user_input` is available, call
it with 1-3 structured questions. If it is unavailable, stop at Stage 2, state
the blocking variable and ask concise text questions. Do not continue to Stage 3
until the user answers.

If no question is `user-private-blocking`, write `无阻塞校准` with the default
judgment and a compact question ledger summary. "No blocking question" never
means "no question"; it means every material question has been self-answered,
assigned to evidence, handled by assumption, or carried as follow-up risk.
Then proceed to Stage 3. Do not manufacture a user question to satisfy the
framework, and do not hide unresolved questions.

### 为什么这个阶段很关键

阶段 1 的分析再全面，也是基于已知信息的推理。而真实决策中，"不知道自己不知道什么"往往是最大的风险。这个阶段的目的是把那些隐藏的决定性因素挖出来。

### 执行要求

1. 先审计阶段 1 的无知地图和可校准模型，不从零临时想问题。
2. 为关键未知项生成问题账本；每个问题必须绑定来源、反转条件、会改变的结论/边界，以及处理动作。
3. 先治理问题，再问用户。上下文、文件、命令、网络事实、常识和 Codex 自己能判断的信息不得问用户。
4. 只把 `user-private-blocking` 问题交给用户：真实目标、价值排序、现实约束、授权边界或外部状态。
5. 禁止责任转移问题：不要问"你选 A 还是 B"、"你更倾向哪个"、"要不要我做"来替代自己的判断。除非 A/B 的区别本质上是用户价值排序，否则 Codex 必须先给默认判断。
6. **引导式提问**：每个用户问题必须带默认判断、理由和反转关系，让用户校准模型，而不是从零替 Codex 思考。

   示例对比：
   - 差：「你想选 A 还是 B？」
   - 好：「我当前判断 A 优于 B，因为 A 更符合 [评判标准]。只有当你的真实优先级是 [Y] 时，排序会反转为 B。这里的真实优先级是 Y 吗？」

7. **阻塞式校准必须使用 Codex 的结构化提问工具**：如果工具可用，使用 `request_user_input`。如果工具不可用，停止在阶段 2，说明观察到的工具/模式证据，并用简短文字提问；不得继续阶段 3。
8. 为了尽可能高质量使用 `request_user_input`，只把 `user-private-blocking` 问题压缩成 **1-3 个阻塞问题**；每个问题给出 2-3 个互斥选项，并把推荐选项放第一位。每个问题必须包含稳定的 `id`、12 字以内的 `header`、以及带 `(Recommended)` 的推荐选项标签。
9. 如果没有阻塞问题，明确写出 `无阻塞校准`、默认判断、问题账本摘要和剩余风险，然后直接进入阶段 3。不得为了使用工具制造问题，也不得跳过问题治理。
10. **一旦提出阻塞问题，必须等待用户回答后才能进入阶段 3**——这是阻塞性的。

### 输出结构

```
### 阶段 2：反向询问

先做校准审计：

**默认判断：** [阶段 1 的当前结论]
**关键假设：** [支撑默认判断的假设]
**反转条件：** [会改变结论/排序/边界的条件]
**可自行判断：** [Codex 应自己查证或推断的信息]
**用户私有变量：** [只有用户能确认且会改变结论的变量；没有则写"无"]

**问题账本：**
- [问题1] | 来源: [未知/反转变量] | 类型: [self-answerable/evidence-needed/assumption-continue/user-private-blocking/follow-up-risk] | 会改变: [结论/边界] | 处理: [自答/查证/假设继续/阻塞提问/后续风险]
- [问题2] | ...

阻塞情况（存在用户私有反转变量）：

基于阶段 1 的可校准模型，我需要先校准一个会改变结论的变量：

1. [问题1]
   → 当前默认判断是 [X]，因为 [A/B/C]。只有当 [Y] 成立时，结论会改成 [Z]。请确认 [Y] 是否成立。

非阻塞情况：

**无阻塞校准：** 阶段 1 没有发现必须由用户立即确认的反转变量；关键问题已通过 [自答/查证/假设/后续风险] 处理，采用 [默认判断/明确假设]，下面直接进入阶段 3。
```

## 阶段 3：计划制定

目标：综合所有信息，在明确边界下作出可验证承诺，给出经过校准的最优方案。

### 执行步骤

**第一步：方案综合**

结合阶段 1 的多路径探索，以及阶段 2 用户确认的信息或明确假设：
- 如果存在一个明显最优方案 → 直接给出该方案及理由
- 如果存在多个各有优劣的方案 → 同时列出，并给出各自特点、适用场景和取舍建议
- 如果用户的回答改变了之前的假设 → 明确说明调整了什么以及为什么

**第二步：方案验证（Reflexion）**

在定稿之前，对方案做一次自检：
- **逻辑一致性**：方案的各个步骤之间是否有矛盾？
- **完备性**：是否覆盖了阶段 1 识别的所有关键维度？
- **可行性**：在用户确认的约束条件下，方案是否真的可执行？
- **风险覆盖**：阶段 1 中识别的风险，方案中是否有对应的应对措施？

如果自检发现问题，在输出中说明并调整，而不是假装没看到。

**第三步：输出方案**

方案必须包含：具体步骤、预期效果、风险及应对。对于执行类任务，步骤应该具体到可以直接执行。

阶段 3 不是分析终点，而是把认识放回实践中验证。方案必须说明如何判断它
有效、失败时说明哪层判断可能错了，以及后续应回到哪一层修正。

阶段 3 必须继承阶段 2 的问题治理结果：
- 哪些问题已经由 Codex 自答或证据处理，
- 哪些假设支撑当前承诺，
- 哪些问题是后续验证风险，
- 哪些条件会推翻当前方案。

不要在阶段 3 假装未知项不存在。承诺必须建立在已治理的问题账本上。

### 输出结构

单一方案时：
```
### 阶段 3：计划

**推荐方案：** [方案名称]
**核心理由：** [为什么是最优——关联到阶段 1 的评判标准]

**具体步骤：**
1. [步骤1] — [预期结果]
2. [步骤2] — [预期结果]
...

**预期效果：** [描述]
**风险及应对：**

**问题治理结果：** [已自答/已查证/假设继续/后续风险/用户已确认的关键问题]
**方案自检：** [一两句话说明自检结果，或说明做了哪些调整]
```

## 阶段 4：实施（仅执行类任务）

目标：按最优方案执行，**并在过程中保持反馈循环**。

### 执行要求

1. 选择阶段 3 中的最优方案开始实施。如果阶段 3 给出了多方案且用户未明确选择，选择综合最优的方案并说明原因
2. **分步交付，边做边报告**：不要等全部做完再一次性展示。每完成一个关键步骤，简要说明进展和发现。这样做有两个好处：用户可以尽早发现偏差，避免大量返工；遇到意外情况时可以及时调整方向
3. 实施过程中遇到预期外问题时，及时说明并调整方案——不要硬着头皮按原计划走
4. 实施完成后进行**结果验证**：
   - 对照阶段 1 的评判标准，检查结果是否达标
   - 对照阶段 3 的预期效果，检查实际效果是否匹配
   - 如有偏差，说明原因和后续建议

## 原则与注意事项

- 每个阶段的输出清晰标注阶段编号和名称
- 显式三步分析调用必须执行阶段 2 问题治理；问题治理不是自动提问。所有关键问题必须有去向，只有阶段 1 的可校准模型证明存在用户私有反转变量时，才提出阻塞式校准问题并等待用户回复
- **不要跳过辩证检验**——它是区分浅层分析和深度分析的关键
- **不要跳过可校准模型和无知地图**——没有默认判断、方案排序、关键假设、反转条件、未知变量和用户私有变量审计，就没有资格进入阶段 2
- **不要跳过问题账本**——真实问题一定有未知项；没有阻塞用户问题不等于没有问题，只表示所有关键问题已有处理动作
- 好问题必须服务于已有判断。不能问没有反转条件的问题，不能问 Codex 自己应该判断的问题，不能把方案选择责任转移给用户
- 分析要有深度，但深度不等于冗长。说到点子上比面面俱到更重要
- 如果在后续阶段发现前面阶段的分析有误或不完整，**允许回溯修正**，并明确说明修正了什么。线性推进不是目的，高质量结论才是
- 输出格式是指引而非枷锁。如果某个问题的特性不适合上面的模板，可以灵活调整结构，但核心步骤（材料铺开→抽象提炼→多维分析→多路径探索→辩证检验→可校准模型/无知地图→问题治理→方案验证）不能省略
