# CO-STAR 结构化任务定义框架

CO-STAR 是快速任务结构化的唯一认知内核。它用六个维度把模糊输入转成可执行规格，而不是增加额外展示流程。

## 软件项目边界

CO-STAR 只负责快速澄清任务表达，不负责软件项目生命周期编排。
当任务涉及软件项目、新项目、架构、开发闭环、发布、交接或多 skill 调用时，
先用 `project-brief` 作为适配层澄清项目意图，或直接用 `project-lifecycle` 编排。
适配层只能加入项目范围、验证和调用链，不另造一套任务结构化哲学。日常写作、
prompt shaping、非项目产出规格才直接使用本 skill。

## CO-STAR 六维度

### C - Context（上下文）
提供任务的背景信息，让 Codex 理解全貌。

回答这些问题：
- 这个任务的背景是什么？为什么现在要做这件事？
- 有哪些相关的前置信息 Codex 需要知道？
- 有没有之前的尝试或已有的材料？

**示例：**
- 差：「写一封邮件」
- 好：「我们公司计划在 2026-05-22 上线新版客户管理系统，需要提前通知所有老客户系统迁移事宜。旧版系统自 2023-05 起运行，这次是大版本升级。」

### O - Objective（目标）
明确你希望 Codex 完成什么，以及什么样的结果算"好"。

回答这些问题：
- 具体要产出什么？（一封邮件？一份方案？一份检查清单？）
- 成功的标准是什么？
- 有没有必须包含或必须避免的内容？

**示例：**
- 差：「通知客户」
- 好：「写一封系统迁移通知邮件，需要：(1) 告知迁移时间和影响范围，(2) 说明新系统的主要改进，(3) 提供数据迁移指引，(4) 给出技术支持联系方式」

### S - Style（风格）
定义产出物的写作风格或表达方式。

考虑这些方面：
- 参考哪种写作风格？（简洁直接如Amazon内部备忘录、叙事型如TED演讲、学术型如论文……）
- 技术深度如何？（面向专家的技术文档 vs 面向小白的入门指南）
- 是否有需要模仿的范例？

### T - Tone（语气）
设定情感基调和沟通态度。

常见语气选项：
- 正式 / 半正式 / 轻松随意
- 热情鼓励 / 客观中立 / 严肃专业
- 亲切友好 / 权威可信 / 幽默风趣

**语气和风格的区别**：风格是"怎么写"（结构、用词复杂度），语气是"什么感觉"（情感色彩）。同样的风格可以有不同的语气——比如同样是简洁直接的风格，可以是温暖的，也可以是冷峻的。

### A - Audience（受众）
明确目标读者/用户是谁，这直接决定了内容的深度和表达方式。

考虑这些方面：
- 他们的专业水平如何？（领域专家 / 有一定背景 / 完全外行）
- 他们关心什么？痛点是什么？
- 他们会在什么场景下阅读/使用这个产出？
- 有多少时间？（忙碌的高管 vs 有时间细读的研究人员）

### R - Response Format（响应格式）
指定输出的具体格式和结构。

常见格式选项：
- 纯文本 / Markdown / HTML / JSON
- 分段落 / 分要点 / 表格对比
- 指定长度（如：300字以内、一页A4纸）
- 是否需要特定结构（如：执行摘要→详细分析→建议）

## 执行流程

### 步骤 1：信息收集

检查用户提供的信息，判断六个维度中哪些已知、哪些缺失。

**对于已知维度：** 直接采用用户提供的信息。
**对于缺失维度：** 根据上下文做出合理推断，并在输出中标注"[推断]"，让用户有机会修正。

判断原则：如果推断的把握度高（比如给CEO写邮件，语气显然应该正式），直接采用推断并执行。如果有多个合理选择且会明显改变产出，才快速确认关键分叉。

### 步骤 2：内部任务卡片

将收集到的信息整理为内部结构化卡片。默认不展示完整卡片；只有用户要求结构化 prompt、任务定义、CO-STAR 卡片，或你需要用户确认关键推断时才展示。

需要展示时使用：

```
### CO-STAR 任务定义

| 维度 | 定义 |
|------|------|
| **C** 上下文 | [背景信息] |
| **O** 目标 | [具体目标和成功标准] |
| **S** 风格 | [写作/表达风格] |
| **T** 语气 | [情感基调] |
| **A** 受众 | [目标读者/用户] |
| **R** 格式 | [输出格式要求] |

[如有推断项，在表格后注明：*标注 [推断] 的部分是我基于上下文的判断，如有偏差请告知。*]
```

### 步骤 3：执行产出

基于内部 CO-STAR 卡片，直接生成产出物。

执行时注意：
- 产出物的每个部分都应该能追溯到CO-STAR的某个维度
- 如果在执行过程中发现任务定义有矛盾（比如要求轻松语气但受众是法律专业人士），主动指出并建议调整
- 产出完成后，进行一次快速自检：是否满足了O（目标）中定义的所有成功标准

### 步骤 4（可选）：快速迭代

如果用户对产出有修改意见：
1. 判断是哪个维度需要调整
2. 更新内部 CO-STAR 卡片中的对应维度
3. 基于更新后的规格重新生成

这比每次从头理解用户意图要高效得多——修改有了明确的"坐标系"。

## CO-STAR 的组合使用

CO-STAR 可以和其他框架组合：

- **CO-STAR + Self-Refine**：先用 CO-STAR 清晰定义非项目产出，再用 Self-Refine 迭代打磨。适合商业提案、公开演讲稿、邮件等。
- **CO-STAR + 三步认真分析**：如果非项目任务需要先分析（比如"写一份竞品分析报告"），先用三步分析做深度分析，再用 CO-STAR 定义写作规格。
- **项目任务**：使用 `project-brief` 定义项目规格；需要深度技术判断时使用 `project-analysis`；项目产物打磨使用 `project-refine`。

## 适用场景速查

| 场景 | 是否适用 CO-STAR | 说明 |
|------|:---:|------|
| 写邮件/通知/公告 | 非常适合 | CO-STAR 的经典场景 |
| 写博客/文章/文案 | 非常适合 | STAR四个维度尤其关键 |
| 整理非项目 prompt/任务说明 | 一般适合 | Context 和 Objective 最重要；项目任务走 project-brief / project-iteration |
| 翻译/润色 | 一般适合 | Style和Tone维度最关键 |
| 非项目深度分析/决策 | 不太适合 | 用三步认真分析更好 |
| 项目深度分析/决策 | 不适合 | 用 project-analysis |
| 简单问答 | 不需要 | 直接回答更高效 |

## 原则与注意事项

- CO-STAR是"快速结构化"工具，执行应该迅速。如果发现自己在 CO-STAR 上花了太多时间思考，说明任务可能更适合 `three-step-analysis`、`project-brief` 或 `project-analysis`
- 六个维度不是每个都同样重要。根据任务类型灵活调整权重——写固定格式文本时R（格式）很重要，写邮件时T（语气）很重要
- 如果用户提供了非常详细的需求，不需要刻板地把信息重新填入六个维度——直接执行即可。CO-STAR是帮助理清思路的工具，不是增加流程的负担
